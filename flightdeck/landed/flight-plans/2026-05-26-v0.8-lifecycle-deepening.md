## Task 1: OpenSpec delta markers in spec template

Add a convention to `skills/workshop-workflow/templates.md` (likely as a new sub-section or addition to existing spec/board guidance) for marking spec evolution with `ADDED:` / `MODIFIED:` / `REMOVED:` prefixes inside change-tracking sections of long-lived specs.

The convention applies primarily to **backlog specs** (specs that live a long time and gain items over multiple sessions) and **revised specs** (specs that get amended after critique). Per the v0.6→v1.0 roadmap example: when GPT critique disposition lands, the roadmap itself was amended — those amendments would carry `MODIFIED:` markers if v0.8 had been in effect.

Convention text to add (sample location: new "spec evolution markers" subsection in templates.md, OR append to the existing scar/playbook lifecycle rules):

```markdown
## Spec evolution markers (optional convention)

When amending a long-lived spec (especially backlog specs in `specs/` or specs revised after critique disposition), mark new/modified/removed items with prefix tags so the change history is grep-able and merge-friendly:

- **`ADDED:`** — new item or section
- **`MODIFIED:`** — existing item changed (note the old + new state if useful)
- **`REMOVED:`** — item dropped (typically commented out or struck-through, not deleted, so the audit trail survives)

Example (from a revised spec):
```
- ADDED: v0.6 #5 wip TTL hard gate
- MODIFIED: v0.7 #1 — emitter now prefixes relative links with `workshop/`
- REMOVED: ~~v0.8 cold-start benchmark~~ (rejected — S1 covers it qualitatively)
```

Markers are **optional** — small one-shot specs don't need them. The benefit kicks in for specs revised across multiple sessions or critiques.
```

## Task 4: Hard-gate wip writes via per-file checklist

Update `skills/workshop-workflow/templates.md` wip template (added in v0.6 Task 5) with a checklist that the AUTHOR must work through before writing a new wip file. Per the v0.6→v1.0 roadmap v0.8 item 4:

> Hard-gate `wip/` writes via per-file checklist in `templates.md`. Author must answer (a) is this surviving past this session? (b) what's the destination? before any new `wip/` file.

Add a "Pre-write checklist" subsection to the wip template:

```markdown
### Pre-write checklist

Before creating any new `wip/` file, the author (human or AI) must answer:

1. **Will this content survive past this session?** If yes — gate this; the right destination is probably `scars/`, `playbooks/`, `specs/`, `sketches/`, or `critiques/`, not `wip/`. If no — proceed to next question.
2. **What's the destination if it does survive?** Either: (a) "delete at session-exit" — confirm intent, or (b) name the destination folder upfront (e.g., "this will graduate to `scars/` if it crystallizes").

If either answer is ambiguous, do NOT create the wip file. Either classify directly, or defer the decision (don't accumulate junk in `wip/`).
```

## Task 5: `/workshop:doctor` slash skill

Create `skills/doctor/SKILL.md` — markdown checklist that audits a workshop for protocol drift. Frontmatter with `disable-model-invocation: true`. Audits to perform:

1. **Scars/playbooks frontmatter present**: every file under `scars/` and `playbooks/` carries `when_to_read`, `applies_to`, `last_updated`.
2. **Stale wip files**: any `wip/*.md` whose `last_touched:` predates the most recent commit by > 7 days, or whose `last_touched:` is missing.
3. **Dangling references**: links in `board.md`, specs, plans, playbooks, scars pointing to files that don't exist.
4. **Orphan scars**: scar files with `status: superseded` but no project-rules upgrade reference; or `status: obsolete` with `last_updated` recent enough to suggest they're not actually obsolete.
5. **Board ↔ folder lifecycle mismatch**: artifacts in `*/finish/` whose board row still says `awaiting-review`; or `state: blocked` artifacts whose board row is missing.
6. **Stale Blockers entries**: per the v0.7 final-review follow-up, check `board.md` Blockers section for entries that reference shipped versions or completed work.
7. **`Recently finished` length**: more than 5 entries.
8. **AGENTS.md regeneration drift**: if `AGENTS.md` exists with workshop markers, check that the workshop block content matches the current state of `board.md`. If not, the emitter hasn't been re-run since board changed.

The skill outputs a structured report with severity (CRITICAL / WARNING / INFO) per finding. Implemented as a slash skill (markdown checklist that the AI follows), NOT a CLI binary, to preserve the plain-markdown bet.

Dogfood: run the new `/workshop:doctor` skill against this repo's own `workshop/` and report findings (expectation: should produce a clean or near-clean report given we just shipped two themed releases).
