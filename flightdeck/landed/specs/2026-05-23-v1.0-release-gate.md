## Scenarios

### S1. Cold-start in a project with workshop/

**Setup**: Fresh subagent given a project that has `workshop/board.md` populated and `workshop/scars/`. Ask: "What should we work on?"

**RED expectation (without skill)**:
- Agent ignores `board.md`, asks user open-ended question.
- Or browses code at random.
- Or attempts a `find . | head` to explore.

**GREEN target (with skill)**:
- Agent reads `board.md` first.
- Reconciles against `git branch --show-current` and `git status`.
- Reports state back to user, then executes the first "next session" item.

**Pass criteria**: agent's first action is `Read workshop/board.md`. Second action is a reconciliation check. Third action is execution.

---

### S2. Bug + root cause discovered mid-session

**Setup**: Subagent debugs a test failure, finds the root cause (e.g., "I assumed the API returned a list but it returns a generator"). Asks: "We're wrapping up — anything to record?"

**RED expectation (without skill)**:
- Agent writes a long debug log inline into `board.md`.
- Or writes a scar file with root cause "I was careless / forgot to check".
- Or skips writing anything.

**GREEN target (with skill)**:
- Agent classifies → `scars/<topic>.md`.
- Uses the scar template; root cause is "I assumed X but actually Y" (not "forgot").
- Adds Status: active.
- Checks if a related scar already exists; if so, appends `## [Case N]`.

**Pass criteria**: scar file created with banned root-cause language absent. Status field present.

---

### S3. Ambiguous knowledge — scar or playbook?

**Setup**: Subagent learns "every time we add a new field, we have to run X, Y, Z or it breaks". Asks the agent to record this.

**RED expectation (without skill)**:
- Agent writes to whichever folder it picks first, without reasoning.
- Or writes to multiple folders (duplication).

**GREEN target (with skill)**:
- Agent recognizes "every time" → `playbooks/` heuristic.
- Or if classification is genuinely ambiguous (matches multiple heuristics), invokes the AI-asks-user template from `exit-ritual.md`.

**Pass criteria**: agent applies first-match-wins heuristic, or asks user with structured options.

---

### S4. wip file from last session

**Setup**: Project has `workshop/wip/2026-05-19-debug-notes.md` (5 days old). Agent is about to close current session.

**RED expectation (without skill)**:
- Agent leaves the wip file untouched.
- Or moves it to `sketches/` reflexively.

**GREEN target (with skill)**:
- Agent identifies stale wip.
- Either classifies into another folder (applying heuristics) or deletes.
- Default is deletion if no clear classification fits.

**Pass criteria**: no wip files older than current session remain after exit.

---

### S5. Critique without disposition

**Setup**: User pastes a long review from another AI. Subagent is asked to "save this critique".

**RED expectation (without skill)**:
- Agent saves only the raw text into a new critique file, no disposition.
- Or summarizes the feedback without acting on each point.

**GREEN target (with skill)**:
- Agent saves raw text + opens a Disposition section.
- If unable to dispose immediately, adds a hanging task to `board.md`: `- [ ] Finish disposition of [critiques/<file>](critiques/<file>)`.
- Refuses to close the session cleanly until hanging task is acknowledged.

**Pass criteria**: critique file has both Raw and Disposition sections (Disposition can be a placeholder + board hanging task).

---

### S6. Default-brainstorm trap

**Setup**: Subagent at session exit. Has 4 pieces of new knowledge: 1 clear bug+rootcause, 1 clear procedure, 1 design decision, 1 one-off log.

**RED expectation (without skill)**:
- Agent invokes brainstorming for all 4 items individually.
- Or invokes brainstorming once for the bundle, taking many turns.

**GREEN target (with skill)**:
- Agent classifies directly:
  - Bug+rootcause → `scars/`
  - Procedure → `playbooks/`
  - Design decision → `specs/` (with optional brainstorming for this one)
  - One-off log → no write
- Brainstorming only triggered (at most) for the design decision, not all 4.

**Pass criteria**: ≤ 1 brainstorm invocation, ≥ 3 direct classifications.

## Test methodology

RED-GREEN-REFACTOR via fresh subagents:

- Dispatch a fresh subagent per scenario (no carry-over context).
- Run RED first (no skill in context), capture exact behavior verbatim.
- Run GREEN with skill loaded.
- If GREEN fails, find the rationalization, add an explicit counter, re-run.
- Iterate until bulletproof.

Each test iteration takes ~5–15 minutes. Budget ~2 hours for the full 6 scenarios with 1–2 refactor rounds each.

## Release gate

This skill is **not** production-ready until:

1. All 6 scenarios pass GREEN with fresh subagents.
2. The rationalization table is non-empty (i.e., real loopholes have been found and closed).
3. A real project has used the skill for at least 5 sessions and `workshop/` has not degraded.

Until then: v0.x, pre-release.

## v0.7 manifest schema verification (2026-05-26)

Schema-level verification of all 5 plugin manifests against each host's documented manifest spec. This is **schema-level only** — behavioral verification (S1–S6 on each tool) requires running each AI tool against a project with `workshop/` and is tracked separately in `.github/PULL_REQUEST_TEMPLATE/manifest-verification.md`.

All manifests currently report `version: 0.6.0`. Version bump to `0.7.0` happens in v0.7 Task 6 (release).

### Claude Code — `.claude-plugin/plugin.json` + `.claude-plugin/marketplace.json`

- **Status**: ✅ matches schema
- **Schema doc**: https://code.claude.com/docs/en/plugins (general) and `/en/plugins-reference` (full schema, not fetched directly)
- **Required**: `name` ✓. `version` is documented as optional (commit SHA used as fallback) but we set it explicitly ✓.
- **Optional present**: `description`, `author`, `homepage`, `repository`, `license`, `keywords`, `skills` (path) — all standard.
- **`skills` field**: docs describe `skills/` as a default-discovered directory; explicit `"skills": "./skills/"` is harmless but redundant since Claude Code auto-discovers `skills/` at the plugin root. Not flagged as wrong.
- **Marketplace**: schema mirrors plugin entries inside `plugins[]` array. Our entry has `name`, `description`, `version`, `source`, `author` — matches the inline plugin metadata pattern used in the community catalog.
- **Behavioral status**: verified by maintainer (this is the home host).

### Codex CLI — `.codex-plugin/plugin.json`

- **Status**: ❓ schema doc unavailable; relying on existing manifest as the working spec — behavioral verification PR required
- **Schema doc fetched**: https://developers.openai.com/codex/guides/agents-md — explicitly states Codex CLI customization is via `AGENTS.md` only. **No `plugin.json` / `.codex-plugin/` schema is publicly documented.**
- **Implication**: our `.codex-plugin/plugin.json` is currently unrecognized by Codex CLI. The functional integration path on Codex is via `AGENTS.md` at project root (shipped in v0.7 Task 1 by `emit-agents-md`).
- **Action items**:
  - The manifest is harmless (Codex ignores unknown files) but its presence may mislead users into thinking Codex auto-installs workshop as a "plugin".
  - The `interface` block (displayName, capabilities, defaultPrompt, etc.) appears to be aspirational / convention-borrowed. Document this in v0.7 Task 5 README clarification.
  - Behavioral verification on Codex CLI must confirm: does Codex pick up `AGENTS.md` emitted by `/workshop:emit-agents-md`? Open question for the community PR.

### Cursor — `.cursor-plugin/plugin.json`

- **Status**: ⚠️ missing optional fields / minor mismatch
- **Schema doc**: https://cursor.com/docs/plugins + https://cursor.com/docs/reference/plugins
- **Required**: `name` ✓ (only required field per spec).
- **Optional present**: `description`, `version` (semver), `author`, `homepage`, `repository`, `license`, `keywords`, `skills` (path string — valid per schema, which accepts string or array).
- **Deltas**:
  - `displayName` (top-level): **not documented in Cursor schema**. Currently set to `"Workshop"`. Cursor likely ignores unknown fields silently, but this is a leftover from a different host's convention. Candidate for removal in a future cleanup (not this task).
  - Cursor schema additionally supports `rules`, `agents`, `commands`, `hooks`, `mcpServers`, `logo` — none currently used. Workshop is skills-only, so no action needed.
  - Submission checklist item: "YAML frontmatter metadata on rules/skills/agents/commands" — already addressed in v0.7 Task 2 (Cursor MDC frontmatter on scar/playbook templates, commit `1a8fb00`).
- **Behavioral status**: unverified. Open for community PR.

### Gemini CLI — `gemini-extension.json`

- **Status**: ✅ matches schema (minimal-but-valid)
- **Schema doc**: https://raw.githubusercontent.com/google-gemini/gemini-cli/main/docs/extensions/reference.md
- **Required**: `name` ✓, `version` ✓, `description` ✓.
- **`contextFileName`**: documented to default to `GEMINI.md` if omitted; accepts any filename string. We set `"contextFileName": "GEMINI.md"` explicitly — redundant but not wrong. Note: spec confirms `AGENTS.md` is a permitted value, so a future change to point Gemini at the same `AGENTS.md` emitted by `emit-agents-md` is viable (would deduplicate context files).
- **Not present (intentionally)**: `mcpServers`, `excludeTools`, `settings`, `plan`, `themes`, `migratedTo`. Workshop is a context-file-only extension; none apply.
- **Not supported by schema**: `author`. Currently absent from our manifest — correct.
- **Behavioral status**: unverified. Open for community PR.
