# v0.6 → v1.0 roadmap

**Date**: 2026-05-25
**Origin**: brainstorm session — market survey (AGENTS.md, Cline Memory Bank, OpenSpec, Roo Boomerang, Cursor MDC, Superpowers v4.3.0, Letta Code, Context Capsule) + Explore-agent project review
**Status**: v0.6 / v0.7 / v0.8 sections shipped; v1.0 section pending dogfood signals
**Applies to**: workshop-project meta-evolution (versioning, format freeze, tool reach)

## Goal

Take workshop from v0.5.0 to v1.0 through one breaking-change window, then open a capability-expansion window post-1.0. By v1.0, file formats and folder semantics are locked; format changes after v1.0 require a major bump.

## Versioning principles

1. **v1.0 = breaking-change freeze.** Renaming files, restructuring folders, changing frontmatter schema all happen pre-1.0.
2. **One theme per pre-1.0 release.** v0.6 = cleanup, v0.7 = portability, v0.8 = lifecycle depth. Mixing themes muddies the changelog and prevents focused dogfooding.
3. **Dogfood ≥ 1 week per release before next.** v0.4 changelog established this rhythm; "1 week of real use" surfaces real defects.
4. **Post-1.0 = additive only.** New capabilities never replace v1.0 contracts.

## Architecture decisions inherited from v0.5

- Authority order: project-rules > board > active-plans > active-specs > playbooks > scars > archived.
- Semi-implicit lifecycle: location = source of truth; frontmatter `state:` overrides for special states.
- Write gate first-match-wins heuristics (a)–(h).
- SessionStart hook auto-injects `workshop-workflow` skill when `workshop/` exists in cwd.
- Workshop dogfoods itself: this repo's `workshop/` is scoped via `applies_to` to workshop-project paths only.

**Theme**: ship the contract.

## Open questions (defaults chosen; pending user confirmation)

1. **Release cadence**: target one pre-1.0 release per week. Default: v0.6 by 2026-06-01, v0.7 by 2026-06-08, v0.8 by 2026-06-15, v1.0 by 2026-06-22.
2. **Codex / Cursor / Gemini access**: maintainer's tool inventory unknown to this spec. Default: v0.7 ships with whatever 2-of-3 the maintainer can verify; the third gets a community-PR slot.
3. **AGENTS.md emitter ordering**: keep in v0.7 (not promoted to v0.6). Default chosen because v0.6's theme is "cleanup only" — mixing in a new surface dilutes it.

## Risks

| Risk | Mitigation |
| --- | --- |
| Format freeze at 1.0 premature; downstream pain forces v2.0 quickly | Use v0.8 to deliberately stress the format (delta markers, MDC frontmatter) so schema weaknesses surface pre-freeze. |
| AGENTS.md emitter overlaps with non-workshop AGENTS.md tooling on the same repo | Implement as additive via fenced markers; never overwrites user-authored content. |
| Dogfooding workshop on itself creates recursive confusion in scars | `applies_to:` scoping is the guardrail. Document explicitly in `workshop/board.md` footer and `README.md` Dogfooding section. |
| v0.7 manifest validation stalls; ecosystem bet unproven | If ≥ 1 of 3 non-Claude tools is verified GREEN, v1.0 ships with that subset (community-PR slots open for the rest). If **all 3** are unverified at v0.7 close, v1.0 freeze is **postponed** until ≥ 2 of 3 are GREEN — workshop's multi-tool claim is the headline value and cannot ship unsupported. |

## Non-goals

- Replacing AGENTS.md ecosystem. workshop emits TO AGENTS.md; it does not try to be AGENTS.md.
- Web UI / dashboard. Plain-markdown + git is the bet.
- LLM-based auto-classification of new knowledge. First-match-wins heuristics stay deliberately mechanical.
- Cross-project memory sharing. Each project's `workshop/` is independent by design.

## Success metrics (post v1.0)

- A fresh AI session in any of the 4 supported tools, given a project with `workshop/`, reads `board.md` first and follows the protocol without intervention.
- A maintainer who steps away for a week resumes by reading only `board.md`.
- 3+ external projects (not the maintainer's) using workshop with `workshop/` committed.
